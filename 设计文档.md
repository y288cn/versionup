# versionup —— 桌面软件工具版本升级公共包

> 纯 Go 实现的轻量级、可嵌入的版本升级公共库。客户端周期性向服务端询问工具最新稳定版本，与本地版本比对，发现差异时发出**版本升级事件**，并提供事件上报、通知管理、定时通知与版本下载能力。可作为 Go module 开源发布，随时 `import` 到任意项目中。

---

## 1. 项目背景与目标

桌面软件（如 github-speed、LinkNest 等）普遍需要「自动检查更新」能力，但各项目重复造轮子：版本协议不统一、比对逻辑散落、事件/通知/下载各自实现。

**目标**：沉淀一个**与业务无关**的通用版本升级包 `versionup`，统一解决：

- 周期性检查远端最新稳定版本
- 语义化版本比对
- 升级事件（含下载地址、版本号、工具 ID 等）的产出与分发
- 升级事件上报（遥测/统计）
- 通知的开启/关闭与指定时间通知
- 版本包下载（带进度、校验）

**非目标（细化）**：不内置「自替换 / 重启 / 提权 / 配置迁移 / 回滚」等平台相关执行逻辑（交给各项目通过 `Applier` 接口注入）。库负责到「把下载到的包交给 `Applier` 应用」为止，并提供跨平台、与业务无关的安全原语（`ExtractArchive` / `RunInstaller` / `VerifyChecksum`）。

## 2. 术语定义

| 术语 | 含义 |
| --- | --- |
| Tool / 工具 | 一个被管理的桌面软件，由唯一 `ToolID` 标识 |
| CurrentVersion | 客户端当前运行的版本号（语义化） |
| Release | 服务端返回的某工具的一个发布版本（含版本号、下载地址、是否稳定等） |
| UpdateEvent | 比对后发现「有新稳定版本」时产出的升级事件 |
| Source | 版本源：拉取某工具最新稳定版的抽象接口 |
| Reporter | 事件上报接口：把升级事件上报到服务端/遥测 |
| Notifier | 通知管理器：控制通知的开关与定时 |
| Downloader | 下载器：把版本包下载到本地 |

## 3. 功能需求（FR）

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-1 | 周期检查 | 客户端按可配置间隔（默认 24h）周期性向 `Source` 询问最新稳定版本 |
| FR-2 | 立即检查 | 提供 `CheckNow` 手动触发一次检查（如用户点击「检查更新」） |
| FR-3 | 版本比对 | 使用语义化版本规则比较 `Latest` 与 `Current`；仅当 `Latest > Current` 且 `Stable=true` 时判定为「需要升级」 |
| FR-4 | 升级事件 | 判定需升级时，构造 `UpdateEvent{ToolID, CurrentVersion, LatestVersion, DownloadURL, ReleaseNotes, PublishedAt, DetectedAt}` |
| FR-5 | 事件分发 | 通过 `Notifier` 把事件推送给所有订阅者（回调/Channel） |
| FR-6 | 事件上报 | 提供 `Reporter` 接口与默认 HTTP 上报实现，将事件上报到服务端统计 |
| FR-7 | 通知开关 | `Enable()` / `Disable()` 控制是否分发通知；`Enabled()` 查询状态 |
| FR-8 | 定时通知 | `NotifyAt(t, event)` 在指定时间投递一次通知（如「凌晨 3 点提醒升级」） |
| FR-9 | 版本下载 | `Download(ctx, url, dest)` 下载版本包，支持进度回调与 SHA256 校验 |
| FR-10 | 可组合 | 所有能力以接口形式暴露，可替换默认实现（如自定义 Source / Reporter） |
| FR-11 | 优雅启停 | `Start(ctx)` / `Stop()` 受 `context` 控制，退出时不泄漏 goroutine |
| FR-12 | 应用更新包 | 提供 `Applier` 接口与跨平台安全原语，把下载到的包按策略应用（安装器/解压/自替换） |
| FR-13 | 解压原语 | `ExtractArchive` 支持 zip/tar/tar.gz/tar.xz，提供保留用户配置的 hook |
| FR-14 | 安装器启动 | `RunInstaller` 启动下载到的安装器（提权/等待由调用方决定） |
| FR-15 | 可选自动应用 | `WithAutoApply(bool)` 控制「发现更新后是否自动调用 `Applier.Apply`」，默认关闭（仅通知） |

## 4. 非功能需求（NFR）

- **纯 Go、零 CGO**：保证跨平台（windows / linux / darwin，amd64/arm64）可直接编译。
- **最小依赖**：仅标准库 + 极少量被广泛信任的库（如语义化版本 `github.com/Masterminds/semver/v3`，可选）。
- **线程安全**：`Updater` / `Notifier` 可被多 goroutine 并发调用。
- **可观测**：关键路径（检查、上报、下载）提供 `log` 接口注入，默认静默。
- **超时与重试**：网络请求带 `context` 超时；检查失败仅记录日志，不 panic。
- **开源友好**：含 `go.mod`、`LICENSE`、`README`、`examples/`，可直接 `go get`。

## 5. 整体架构

```
┌─────────────── 应用进程 ───────────────┐
│                                         │
│   YourApp ──import──> versionup.Updater │
│                              │          │
│        ┌─────────────────────┼──────────┘
│        │                     │
│   ┌────▼─────┐         ┌─────▼──────┐
│   │ Checker  │ 周期    │  Notifier  │ 通知开关/定时
│   │ (定时)   │───────▶ │ (事件总线) │──▶ Observer 回调
│   └────┬─────┘         └────────────┘
│        │ Latest()                │ Report()
│   ┌────▼─────┐            ┌──────▼─────┐
│   │  Source  │            │  Reporter  │
│   │ (HTTP)   │            │ (HTTP 上报)│
│   └────┬─────┘            └────────────┘
│        │ Download()              ▲
│   ┌────▼─────┐                  │
│   │ Downloader│──────────────┐   │
│   └────┬─────┘              │   │
│   ┌────▼─────┐              │   │
│   │  Applier  │ (注入策略)   │   │
│   │ 安装/解压/│◀─────────────┘   │
│   │  自替换   │                  │
│   └──────────┘                  │
│                           版本包 升级事件
└────────────────────────────────────┘
```

数据流：`Checker` 周期调用 `Source.Latest` → 比对 → 若需升级，构造 `UpdateEvent` → 经 `Notifier` 分发给订阅者，并经 `Reporter` 上报；订阅者/应用可调用 `Downloader.Download` 拉取安装包，再经 `Applier`（注入的策略：安装器/解压/自替换）应用更新。

## 6. 接口与核心类型设计

### 6.1 数据模型

```go
// Release 服务端返回的一个稳定发布版本。
type Release struct {
    ToolID       string    `json:"tool_id"`
    Version      string    `json:"version"`       // 语义化版本，如 v1.2.3
    Stable       bool      `json:"stable"`        // 是否稳定版
    DownloadURL  string    `json:"download_url"`  // 安装包下载地址
    ReleaseNotes string    `json:"release_notes"` // 更新说明
    PublishedAt  time.Time `json:"published_at"`
    Checksum     string    `json:"checksum"`      // 可选，sha256 hex
}

// UpdateEvent 版本升级事件。
type UpdateEvent struct {
    ToolID         string
    CurrentVersion string
    LatestVersion  string
    DownloadURL    string
    ReleaseNotes   string
    PublishedAt    time.Time
    DetectedAt     time.Time
}
```

### 6.2 版本源 Source

```go
// Source 版本源：拉取某工具最新稳定版。
type Source interface {
    Latest(ctx context.Context, toolID string) (*Release, error)
}
```

默认实现：
- `NewHTTPSource(baseURL string) Source`：`GET {baseURL}/version?tool={id}` 返回 `Release` JSON。
- `NewManifestSource(url string) Source`：直接拉取一个静态 `version.json` 清单文件。

### 6.3 事件上报 Reporter

```go
// Reporter 版本升级事件上报接口。
type Reporter interface {
    Report(ctx context.Context, event UpdateEvent) error
}
```

默认实现 `NewHTTPReporter(endpoint string) Reporter`：`POST` 事件 JSON 到上报端点（失败仅记日志，不影响主流程）。

### 6.4 通知 Notifier

```go
// Observer 事件观察者（通知订阅者）。
type Observer func(UpdateEvent)

// Notifier 通知管理器：控制通知开关与定时。
type Notifier interface {
    Enable()                        // 开启事件通知
    Disable()                       // 关闭事件通知
    Enabled() bool                  // 查询当前是否开启
    Subscribe(obs Observer)         // 订阅升级通知
    NotifyAt(t time.Time, e UpdateEvent) // 指定时间投递一次通知
    Dispatch(e UpdateEvent)         // 内部：若开启则广播给订阅者
}
```

### 6.5 下载 Downloader

```go
type Downloader interface {
    Download(ctx context.Context, url, dest string, opts ...DownloadOption) error
}

// DownloadOption 支持：
//   WithProgress(fn func(done, total int64))  // 进度回调
//   WithChecksum(sum string)                  // 下载后校验 sha256
```

默认实现 `NewHTTPDownloader() Downloader`，支持断点续传可选、进度、校验。

### 6.6 应用层 Applier（自动更新）

下载到的版本包有三种常见形态，处理方式差异很大，因此库**只定义抽象接口 + 提供跨平台安全原语**，具体「如何应用」由调用方通过 `Applier` 注入，避免把平台相关逻辑固化进公共包。

```go
// Applier 应用（安装）下载到的更新包。具体策略由项目选择。
type Applier interface {
    Apply(ctx context.Context, pkgPath string, ev UpdateEvent) error
}
```

库直接提供的、跨平台且通用的原语（纯 Go，stdlib 即可，零 CGO）：

```go
// ExtractArchive 解压 zip / tar / tar.gz / tar.xz 到 dest 目录。
// 提供 WithFileHook（解压每个条目后回调，返回 ErrSkip 则跳过该条目，用于保留用户配置）等选项。
func ExtractArchive(src, dest string, opts ...ArchiveOption) error

// RunInstaller 启动安装器（case1：.exe/.msi/.pkg/.deb 等），等待其退出。
// 提权、是否等待完成由调用方决定，本函数仅负责启动进程。
func RunInstaller(path string, args ...string) error

// VerifyChecksum 校验文件 sha256 是否与 sum（"sha256:..." 或纯 hex）一致。
// 已由 Downloader 内部使用，这里导出供 Applier 复用。
func VerifyChecksum(path, sum string) error
```

三种形态对应的策略骨架（建议放在项目侧，或作为可选子包 `versionup/apply`；核心库不内置，避免把平台相关逻辑固化）：

```go
// 策略一：安装器（case1）—— 直接启动，后续由安装器自身完成。
type InstallerStrategy struct{ Args []string }
func (s InstallerStrategy) Apply(ctx context.Context, pkg string, ev UpdateEvent) error {
    return versionup.RunInstaller(pkg, s.Args...)
}

// 策略二：压缩包（case2）—— 解压，但通过 hook 保留用户配置、不覆盖。
type ArchiveStrategy struct {
    Dest     string
    Preserve func(rel string) bool // 返回 true 表示保留现有文件（不覆盖）
}
func (s ArchiveStrategy) Apply(ctx context.Context, pkg string, ev UpdateEvent) error {
    return versionup.ExtractArchive(pkg, s.Dest,
        versionup.WithFileHook(func(rel string, r io.Reader) error {
            if s.Preserve(rel) {
                return versionup.ErrSkip // 跳过，保留现有文件
            }
            return nil
        }))
}

// 策略三：单可执行文件（case3）—— 改名+重启，平台相关，最复杂。
// 核心库不实现；建议参考 go-update / Squirrel 等成熟方案，或项目侧实现：
//   1) 下载新 exe 到临时名；2) 将当前 exe 重命名为 .old；
//   3) 新 exe 落到原路径；4) 启动新 exe 并退出自己（Windows 需 move-on-reboot 处理文件锁）。
type SelfReplaceStrategy struct{ CurrentExe string }
func (s SelfReplaceStrategy) Apply(ctx context.Context, pkg string, ev UpdateEvent) error {
    return versionup.ErrNotImplemented // 由项目按目标平台实现（见第 14 节风险与扩展）
}
```

`Updater` 通过 `WithApplier` 注入策略；周期检查/ `CheckNow` 在「需升级」时可选择性自动调用 `Applier.Apply`（由 `WithAutoApply(bool)` 控制，默认 false，即仅通知），或仅把事件交给订阅者由应用决定何时应用。

### 6.7 顶层聚合 Updater

```go
type Updater struct {
    ToolID   string
    Current  string
    Source   Source
    Reporter Reporter // 可选
    Notifier Notifier // 可选
    Checker  *Checker
}

func New(toolID, current string, opts ...Option) *Updater
func (u *Updater) Start(ctx context.Context) error // 启动周期检查
func (u *Updater) Stop()
func (u *Updater) CheckNow(ctx context.Context) (*UpdateEvent, error) // 立即检查
func (u *Updater) OnUpdate(obs Observer)  // 便捷订阅
```

`Option` 模式：`WithSource`、`WithReporter`、`WithNotifier`、`WithInterval`、`WithLogger`。

## 7. 服务端协议（版本清单）

`Source` 默认对接的 HTTP 端点规范：

```
GET {baseURL}/version?tool={toolID}
Accept: application/json
```

响应（200）：

```json
{
  "tool_id": "github-speed",
  "version": "v1.4.0",
  "stable": true,
  "download_url": "https://releases.example.com/github-speed/v1.4.0/github-speed-windows-amd64.zip",
  "release_notes": "修复断网问题，优化首字节延迟",
  "published_at": "2026-07-10T08:00:00Z",
  "checksum": "sha256:9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08"
}
```

- 非稳定版（`stable=false`）视为「不提示升级」。
- 多平台下载：建议在 `download_url` 之外扩展 `assets` 数组（后续版本），本期以单 URL 为主。
- 
- go 1.22兼容性：
- # 每次 commit / 发版前
   go mod tidy
   go build -gcflags="-lang=1.22" ./...
   go vet ./...
   go test ./...

## 8. 版本号规范

采用 [SemVer 2.0](https://semver.org/lang/zh-CN/)：`MAJOR.MINOR.PATCH`，允许前缀 `v`。
比对规则：

- 去除前缀 `v` 后解析为 `semver.Version`。
- 仅当 `Latest > Current` 且 `Stable` 为真时判定需升级。
- 预发布版本（如 `1.5.0-rc1`）按 semver 规则低于正式版，不参与稳定升级判定。

## 9. 典型使用示例

```go
package main

import (
    "context"
    "log"
    "strings"
    "time"

    "github.com/y288cn/versionup"
)

func main() {
    up := versionup.New(
        "github-speed", "v1.3.0",
        versionup.WithSource(versionup.NewHTTPSource("https://api.example.com")),
        versionup.WithInterval(24*time.Hour),
        versionup.WithReporter(versionup.NewHTTPReporter("https://api.example.com/report")),
        // 注入应用策略：压缩包解压并保留用户配置；设为 WithAutoApply(true) 可发现即自动应用
        versionup.WithApplier(versionup.ArchiveStrategy{
            Dest:     ".",
            Preserve: func(rel string) bool { return strings.HasPrefix(rel, "config") },
        }),
    )

    // 订阅升级通知（开启通知是默认行为）
    up.OnUpdate(func(e versionup.UpdateEvent) {
        log.Printf("发现新版本 %s -> %s，下载: %s", e.CurrentVersion, e.LatestVersion, e.DownloadURL)
        // 可选：触发下载
        _ = versionup.NewHTTPDownloader().Download(context.Background(), e.DownloadURL, "update.zip",
            versionup.WithProgress(func(done, total int64) { log.Printf("%.1f%%", float64(done)/float64(total)*100) }))
    })

    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()
    if err := up.Start(ctx); err != nil {
        log.Fatal(err)
    }

    // 定时通知示例：明天 03:00 提醒升级
    up.Notifier.NotifyAt(time.Now().Add(24*time.Hour).Truncate(24*time.Hour).Add(3*time.Hour), /*event*/)

    select {} // 实际项目中由主程序生命周期管理
}
```

## 11. 发布为 Go module（开源）

> 版本号从 **v1.0.0** 开始（首个稳定可用版即 1.0.0，不发布 0.x 预览版），全程遵循 [SemVer 2.0](https://semver.org/lang/zh-CN/)。

### 11.1 仓库与模块初始化

1. **建仓库**：在 GitHub 新建 `versionup` 仓库（公开、勾选 Add a README 可后补），本地 `git clone` 后进入目录。
2. **模块路径**：确定最终 module path（决定别人 `import` 的路径），例如：
   - `github.com/<your-org>/versionup`
   - 个人：`github.com/<your-github-id>/versionup`
   - 执行 `go mod init github.com/y288cn/versionup` 生成 `go.mod`。
3. **Go 版本**：`go.mod` 中 `go 1.21`（或项目实际最低支持版本），保证 `io`/`os/exec` 等特性可用且覆盖面广。
4. **依赖收敛**：`go mod tidy` 后确认仅含必要依赖（标准库 + 可选 `semver`）；`require` 块保持精简，避免传递膨胀。

- 仅改实现、补测试后，**递增版本号打新 tag**：bug 修复 `v1.0.1`、向后兼容新增 `v1.1.0`、破坏性变更 `v2.0.0`（此时需将 module path 改为 `.../versionup/v2` 以兼容 SemVer 的 v2+ 规则）。
- 使用者升级：`go get -u github.com/y288cn/versionup` 或指定版本 `go get github.com/y288cn/versionup@v1.1.0`。
- 破坏性变更前，建议在 README/CHANGELOG 明确迁移说明。

### 阶段一：脚手架与基础类型

1. **初始化模块**
   - `go mod init github.com/y288cn/versionup`，添加 `semver` 依赖（若选用）。

   - 实现 `NeedsUpgrade(current, latest string, stable bool) bool`。

### 阶段二：核心能力接口与默认实现

4. **版本源**（`source.go`）
   - 定义 `Source` 接口。
   - 实现 `httpSource`（`NewHTTPSource`）：`net/http` GET + `json.Decode`，带 `context` 超时。

5. **事件上报**（`reporter.go`）
   - 定义 `Reporter` 接口。
   - 实现 `httpReporter`（`NewHTTPReporter`）：POST JSON，失败仅记日志不返回致命错误。

6. **通知管理**（`notifier.go`）
   - 定义 `Notifier` 接口与 `Observer`。
   - 实现 `defaultNotifier`：`sync.RWMutex` 保护 `enabled` 与 `observers`；`Dispatch` 在 `enabled` 时并发调用各 `Observer`；`NotifyAt` 用 `time.AfterFunc` 在指定时间触发一次 `Dispatch`。

8. **周期检查器**（`checker.go`）

9. **顶层 Updater**（`versionup.go`）
   - 实现 `New(toolID, current, opts...)`、`Option` 集合、`Start/Stop/CheckNow/OnUpdate`。

### 阶段五：应用层（自动更新）

13. **Applier 接口与原语**（`apply.go`）
    - 定义 `Applier` 接口（`Apply(ctx, pkgPath, ev)`）。
    - 实现 `ExtractArchive(src, dest, opts...)`：按扩展名识别 zip / tar / tar.gz / tar.xz，逐条目解压；支持 `WithFileHook`（返回 `ErrSkip` 时跳过该条目，用于保留用户配置）。
    - 实现 `RunInstaller(path, args...)`：用 `os/exec` 启动安装器进程。
    - 导出 `VerifyChecksum(path, sum)`（复用 `Downloader` 的校验逻辑）。

14. **三种策略骨架**（`apply.go`）
    - `InstallerStrategy`：`Apply` 直接调用 `RunInstaller`。
    - `ArchiveStrategy`：`Apply` 调用 `ExtractArchive` + `WithFileHook` 按 `Preserve(rel)` 跳过用户配置。
    - `SelfReplaceStrategy`：仅返回 `ErrNotImplemented` 占位，注释给出 Windows 文件锁处理与自重启的推荐实现路径（依赖 `go-update`/sidecar，不固化进核心库）。

15. **接入 Updater**（`versionup.go`）
    - 新增 `Option`：`WithApplier(a Applier)`、`WithAutoApply(bool)`（默认 false）。
    - 检查流程在「需升级」分支：若 `AutoApply` 为真且 `Applier` 非空，则在下载后调用 `Applier.Apply`；否则仅通知。
    - 注意：自动应用失败仅记日志，不阻塞检查循环。

## 13. 测试要点

- **比对正确性**：`v1.2.0` vs `v1.2.0`（否）、`v1.2.0` vs `v1.3.0`（是）、`v1.3.0` vs `v1.3.0-rc1`（否，预发布低于正式）、`v1.10.0` vs `v1.9.0`（是，按数值而非字典序）。
- **稳定性过滤**：`stable=false` 即使版本更高也不提示。
- **通知开关**：`Disable` 后 `Dispatch` 不触发订阅者；`Enable` 后恢复。
- **定时通知**：`NotifyAt` 在到达时间精确投递一次（测试用秒级延时，避免长等待）。
- **下载校验**：错误 checksum 应返回错误；进度回调被调用且 `done==total`。
- **解压原语**：`ExtractArchive` 对 zip/tar.gz 解压内容正确；`WithFileHook` 返回 `ErrSkip` 时对应条目被跳过（验证配置保留）。
- **安装器启动**：`RunInstaller` 能启动一个假进程并等待退出（用 `exec.LookPath("cmd")` 或写临时脚本验证）。
- **策略骨架**：`InstallerStrategy.Apply` 等价于启动；`ArchiveStrategy.Apply` 解压后用户配置文件未被覆盖；`SelfReplaceStrategy.Apply` 返回 `ErrNotImplemented`。

## 14. 风险与扩展

- **网络不可达**：检查/上报失败仅记日志，不阻塞主流程；可加指数退避重试（本期可选）。
- **服务端协议演进**：`Source` 接口隔离了协议细节，未来扩展 `assets`（多平台）或签名校验不影响调用方。
- **强制升级**：后续可在 `Release` 增加 `force bool`，由调用方决定阻断启动。
- **签名校验**：下载后可增加 GPG/代码签名校验扩展点（`DownloadOption`）。
- **多工具管理**：`Updater` 以 `ToolID` 区分，可在上层用 `map[string]*Updater` 管理多个工具。
- **自替换/重启（平台相关）**：`SelfReplaceStrategy` 不内置。Windows 不能覆盖正在运行的 exe（文件锁），推荐方案：下载新 exe 到临时名 → 将当前 exe 重命名为 `.old` → 新 exe 落到原路径 → 启动新 exe 并退出自己；若仍被锁则用 `MoveFileEx` 的 `MOVEFILE_DELAY_UNTIL_REBOOT` 延迟到重启替换。提权、配置迁移、回滚由项目侧实现，核心库保持轻量。
- **配置保留**：压缩包策略通过 `WithFileHook`+`Preserve` 避免覆盖用户配置；调用方必须显式声明哪些路径受保护。
